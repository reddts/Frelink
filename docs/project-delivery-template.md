# 项目推进模板

## 1. 总目标

- 项目名称：
- 最终目标：
- 不做什么：
- 核心衡量指标：
- 当前阶段：

## 2. 里程碑设计

每个项目建议拆成 `M0-M5` 六层，避免目标过大导致推进失真。

### M0 立项与基线
- 明确目标、边界、非目标
- 明确当前代码/产品/部署现状
- 明确风险项、依赖项、回滚路径

### M1 框架收口
- 稳定主结构
- 明确目录、模块、核心数据流
- 清理与目标冲突的旧路径

### M2 关键工作流打通
- 至少打通一条主链路
- 从“可展示”推进到“可执行”
- 让后台/操作流真正可用

### M3 公共入口成型
- 首页、导航、详情页、移动端统一表达
- 不再依赖隐藏入口或临时页面
- 形成对外稳定可见结构

### M4 冷启动与内容/数据准备
- 种子内容、种子数据、初始配置补齐
- 做最小可用运营闭环
- 完成首轮对外试运行条件

### M5 API / 自动化 / Agent 扩展
- 在框架稳定后再补 API
- 输出调用说明、权限边界、自动化流程
- 避免过早暴露不稳定接口

## 3. 里程碑记录格式

每个里程碑必须同时更新三处：

### `优化计划.md`
- 写目标
- 写阶段
- 写待办
- 写已完成勾选项

### `CHANGELOG.md`
- 写本里程碑完成了什么
- 写影响范围
- 写验证结果
- 写下一步方向

### `git commit`
- 必须中文
- 必须体现里程碑含义
- 推荐格式：
  - `里程碑一：完成首页与主工作流重构`
  - `里程碑二：完成后台运营闭环`

## 5. 每次推进的最小闭环

每一轮推进至少要完成以下闭环：

1. 明确本轮目标
2. 改代码或改文档
3. 做最小验证
4. 同步到服务器
5. 更新计划或日志

如果一次推进不能形成闭环，至少要明确：
- 为什么不能闭环
- 当前卡点是什么
- 下一轮第一步做什么

## 6. 多 Agent 协同规则

### 主 Agent 负责
- 明确本轮唯一主目标
- 划分子任务
- 合并结果
- 统一更新计划、日志、提交

### 子 Agent 负责
- 只做一个边界清晰的子任务
- 不同时修改同一文件
- 返回改动文件、结果、风险

### 协同原则
- 先定主线，再开子任务
- 不允许多个 Agent 同时改同一写入区域
- 主 Agent 统一做最终提交
- 子 Agent 的结果必须进入日志，而不是只存在聊天记录

## 7. 阶段性检查点

每个阶段结束时，至少检查：

- 目标是否仍然一致
- 是否出现方向漂移
- 是否有临时修补未回收
- 是否需要做一次里程碑提交
- 是否需要更新 CHANGELOG 和优化计划

## 8. Definition of Done

一个阶段只有满足以下条件才算完成：

- 代码已经写入仓库
- 必要验证已经完成
- 服务器已同步或明确说明未同步原因
- `优化计划.md` 已更新
- `CHANGELOG.md` 已更新
- 已有中文 commit

## 9. 推荐节奏

### 日常推进
- 小步快跑，但每轮必须闭环

### 阶段推进
- 每完成一个稳定子阶段，提交一次中文 commit

### 里程碑推进
- 每完成一个里程碑，更新 CHANGELOG，并补一条明确的里程碑记录

## 10. 常见失控信号

- 只改页面文案，不改真实结构
- 一直扩接口，但主框架未稳定
- 改动很多，但没有计划更新
- 做了很多轮，但没有 commit
- 有聊天记录，没有项目记录
- 多 Agent 并行后没人统一收口

## 11. 推荐目录

```text
项目根目录/
  优化计划.md
  CHANGELOG.md
  docs/
    project-delivery-template.md
    reform-roadmap.md
    api-v1.md
```

## 12. 使用方式

新项目启动时：

1. 复制本模板
2. 先写目标、里程碑、非目标
3. 第一次推进前先建立 `优化计划.md` 与 `CHANGELOG.md`
4. 从 M0 开始推进，不要直接跳到 API 或自动化
