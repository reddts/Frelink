# Frelink 性能与索引优化计划（2026-03-14）

## 本次检查范围
- 首页：`https://www.frelink.top/`
- 话题页：`https://www.frelink.top/topics/`
- 问题页：`https://www.frelink.top/questions/`

## 实测结果汇总

### 1) 首页（`/`）
- TTFB（多次抽样）：约 `0.82s ~ 1.28s`，个别请求更高
- 页面总耗时（多次抽样）：约 `1.02s ~ 1.49s`
- 已生效：`gzip`、静态长缓存（`max-age=2592000`）
- 发现问题：
  - 首屏同步 CSS 仍较多（7+ 个），存在渲染阻塞链
  - 首页有 20 张图片，`loading="lazy"` 数量为 0
  - 列表图片存在多张原图直出（MB 级），造成首屏后网络拥堵
  - 顶部背景图 `top-img.png` 约 366KB，对 LCP 影响明显

### 2) 话题页（`/topics/`）
- 抽样结果：
  - run1 `ttfb=1.476s total=1.478s`
  - run2 `ttfb=0.810s total=0.811s`
  - run3 `ttfb=1.046s total=1.664s`
- 资源结构：
  - `img_total=5`，`img_lazy=0`
  - `css_links=8`，`js_src=11`，`defer_js=9`
- 发现问题：
  - 首屏图片未启用 lazy（虽然数量少，但策略未统一）
  - 仍沿用较重公共资源包（CSS/JS 组合未按页面最小化）

### 3) 问题页（`/questions/`）
- 抽样结果：
  - run1 `ttfb=0.737s total=1.005s`
  - run2 `ttfb=0.899s total=1.153s`
  - run3 `ttfb=0.807s total=1.177s`
- 资源结构：
  - `img_total=20`，`img_lazy=0`
  - `css_links=8`，`js_src=13`，`defer_js=11`
- 发现问题（高优先级）：
  - 头像 URL 拼接错误：大量出现 `https://www.frelink.topstorage/...`
  - 该地址无法建立连接，触发 `onerror` 回退默认头像，造成大量失败请求和额外等待
  - 图片未启用 lazy，列表滚动区请求压力偏大

### 4) 文章页（`/articles/`）
- 抽样结果：
  - run1 `ttfb=0.963s total=1.719s`
  - run2 `ttfb=0.749s total=0.920s`
  - run3 `ttfb=1.186s total=1.568s`
- 资源结构：
  - `img_total=20`，`img_lazy=0`
  - `css_links=8`，`js_src=13`，`defer_js=11`
- 发现问题（高优先级）：
  - 文章列表图片存在原图直出（多张 `1MB~3MB+`）
  - 首屏外图片仍立即请求，明显拉高总下载时间

### 服务器同步与验证规则
- 从现在开始，每次明确代码改动完成后，必须同步到服务器再结束本轮。
- 不允许只在本地完成修改而不做服务器端验证。
- 本地验证通过不等于完成，远程服务器验证通过才算本轮真正闭环。
- 文档、模板、配置、脚本等只要影响真实发布流程，也按“必须上传服务器验证”处理，不能以“不是 PHP 代码”为由跳过。
- [x] 已建立本地标准部署环境：
  - 统一入口：`bash scripts/deploy.sh [show-config|sync|verify|deploy|shell]`
  - 本地私有配置：`deploy.local.json`（已加入 `.gitignore`）
  - 同步排除规则：`.deployignore`
- 服务器端最小验证至少包括：
  - 相关 PHP 文件 `php -l`
  - 缓存清理或必要的框架刷新命令
  - 本轮改动涉及页面或接口的最小可用性检查
- 若因权限、网络、环境异常等原因未能同步或验证，必须在当轮记录具体阻塞原因，不能省略。
- `CHANGELOG.md` 中的验证说明，应优先记录服务器端真实验证结果，而不是只写本地检查结果。

### 远程验证红线
- 任何任务若未完成上传服务器与远程验证，不得：
  - 勾选为已完成

- 后续每轮推进默认流程固定为：`本地修改 -> 上传服务器 -> 远程验证 -> 更新优化计划/CHANGELOG -> commit`

## 多语言适配补充

### 当前问题
- 近期为了快速收敛公开表达，前台模板中新增了一批静态中文文案。
- 这会削弱原模板依赖 `L()` 语言包的多语言能力，导致中文之外的站点表达不完整。
- 当前受影响的主要区域：
  - 首页与导航的新内容类型命名
  - FAQ / 综述 / 观察 / 帮助相关入口
  - 详情页、列表页、发布页中新增的知识系统文案

### 原则
- 后续不再直接在模板里持续扩散静态中文。
- 新增的公开表达优先走统一 helper 或语言包映射，再落到模板。
- 多语言适配分两步做，不和当前结构改革互相打架：
  - 第一步：抽取核心内容类型与导航命名为统一 helper，可被语言包覆盖
  - 第二步：按页面批次回收现有静态中文，优先首页、导航、列表页、详情页

## 内容差异化与运营方向补充

### 定位原则
- 运营方向向 `思想碎片`、`研究综述`、`长期主题追踪` 靠拢，不做泛资讯流。
- 差异化重点放在“个人长期判断、研究路径、资料筛选与脉络整理”，这是当前 AI 最难稳定替代的部分。
- AI 主要负责检索、归档、聚合、选题建议，不替代站长的判断、取舍与结论表达。

### 核心内容栏目建议
- `思想碎片`
  - 记录短判断、问题意识、未完成思路、阅读中的即时标注。
  - 价值不在结论完整，而在于保留“思考发生的现场”。
  - 适合移动端高频更新，也适合作为后续长文和专题的素材池。
- `研究综述`
  - 围绕一个主题，整理背景、关键问题、主要资料、分歧点、当前判断、待验证问题。
  - 重点不是堆资料，而是输出“为什么值得关注”和“接下来该看什么”。
  - 可作为 Frelink 区别于普通博客、问答站和 AI 拼接内容的核心栏目。
- `主题追踪`
  - 对持续变化的主题做阶段性记录，例如某工具、某技术方向、某行业议题。
  - 每次更新不是重写全文，而是补“最近变化、旧判断是否失效、下一阶段观察点”。

### 与当前站点结构的结合方式
- `思想碎片` 不建议单独做一级导航，可放在文章体系内，用统一标签或主题管理。
- `研究综述` 应优先进入文章和专题体系，并绑定主题页，形成长期可检索资产。
- `帮助中心` 继续承接稳定知识；`研究综述` 负责高密度判断；`思想碎片` 负责原始思考与线索积累。
- 推荐内容链路调整为：
  - `思想碎片 -> 研究综述 -> 专题/帮助 -> 相关问题`

### AI 与人工的分工
- AI 负责：
  - 收集最近搜索词、热点内容、资料源
  - 生成提纲、对比清单、时间线草稿
  - 提醒哪些主题值得补综述或做阶段性更新
- 站长负责：
  - 判断哪些问题值得写
  - 确定综述的研究边界与取舍标准
  - 输出个人判断、分歧立场和结论

## 热门但不标题党的内容策略补充

### 原则
- 内容必须同时满足两层目标：
  - `让用户愿意点开`
  - `让用户看完觉得有判断价值`
- 不追求空洞热点，不做纯情绪标题，不做“看似有用但没有结论”的内容。
- 热门不是目标本身，`高兴趣入口 + 高判断密度` 才是 Frelink 的内容方向。

### 推荐的内容主线
- `热点解释`
  - 用当前有关注度的话题切入，但重点解释“为什么重要、影响谁、接下来怎么看”。
- `问题判断`
  - 用用户真实会搜的问题做标题，但正文必须补足边界、取舍和判断标准。
- `研究综述`
  - 整理一个主题的资料脉络、主要分歧和当前判断，形成站点壁垒内容。
- `思想碎片`
  - 记录未成文但有价值的判断、观察和疑问，为后续长文提供原材料。
- `专题收口`
  - 将分散内容重新聚合成系列资产，方便搜索、推荐和持续阅读。

### 发布前筛选标准
- [ ] 用户能否在 `3` 秒内知道这篇和自己有关
- [ ] 用户能否在 `30` 秒内获得一个新判断
- [ ] 用户看完后是否愿意继续点下一篇
- [ ] 标题是否真实反映正文，不靠夸张承诺骗点击

### 不做的内容
- 单纯搬运资讯
- 没有结论的伪综述
- 为了点击故意夸张的标题
- AI 拼接痕迹明显、没有立场和取舍的长文

## 站内内容类型体系与标题模板

### 内容类型体系
- `热点解释`
  - 用途：承接当前关注度和外部流量
  - 目标：解释热点背后的真实问题和影响
- `问题判断`
  - 用途：承接搜索流量和用户焦虑点
  - 目标：回答“怎么判断、什么时候不该这样做”
- `研究综述`
  - 用途：建立长期价值和内容壁垒
  - 目标：输出脉络、分歧、证据和当前结论
- `思想碎片`
  - 用途：保持更新频率和个性表达
  - 目标：沉淀原始判断，供后续整理
- `主题追踪`
  - 用途：持续更新一个不断变化的话题
  - 目标：记录变化、修正旧判断、提出下一步观察点
- `FAQ/帮助`
  - 用途：承接稳定知识和高频基础问题
  - 目标：降低重复解释成本，提升搜索可见度

### 标题模板
- `热点解释`
  - 模板：`这件事为什么值得关注：它真正改变了什么`
  - 模板：`XX 很热，但真正重要的不是功能，而是 YY`
  - 模板：`XX 更新之后，谁会受益，谁会被淘汰`
- `问题判断`
  - 模板：`XX 应该怎么选，关键不在 A，而在 B`
  - 模板：`很多人问 XX，其实真正该先判断的是 YY`
  - 模板：`XX 能不能做，不是看能否实现，而是看是否值得`
- `研究综述`
  - 模板：`关于 XX 的一份研究综述：问题、分歧与当前判断`
  - 模板：`XX 到底发展到哪一步了：资料整理与阶段结论`
  - 模板：`如果只花 20 分钟理解 XX，这些问题最值得先看`
- `思想碎片`
  - 模板：`关于 XX 的一个短判断`
  - 模板：`我最近反复想到的一个问题：XX`
  - 模板：`先记下这个观察：XX 可能不是表面那样`
- `主题追踪`
  - 模板：`XX 追踪第 N 期：最近变了什么`
  - 模板：`上次判断要修正吗：关于 XX 的阶段更新`
  - 模板：`XX 这段时间最值得继续观察的 3 个点`

### 内容模板建议
- `热点解释`
  - 建议结构：事件背景 / 真正问题 / 影响对象 / 我的判断 / 下一步阅读
- `问题判断`
  - 建议结构：问题定义 / 常见误区 / 判断标准 / 推荐做法 / 不适用情况
- `研究综述`

- `思想碎片`
  - 建议结构：观察 / 触发原因 / 暂时判断 / 后续待补资料

## 栏目结构优化补充

### 当前问题
- 当前一级栏目仍混合了“内容类型”“人物入口”“运营入口”，层级不统一。
- `专栏`、`创作者` 更适合作为二级入口或个人中心能力，不适合继续放在公开知识站的一级主导航。
- `问题`、`文章`、`主题`、`专题` 之间的关系还不够清晰，用户不容易理解“先去哪看，再去哪继续读”。

### 建议的一级栏目结构
- `首页`
- `主题`
- `问题`
- `文章`
- `专题`
- `帮助中心`

说明：
- `首页`：搜索入口、热点入口、最近更新与继续阅读。
- `主题`：整个站点的知识索引和聚合中枢，应前置。
- `问题`：承接用户直接搜索和具体问题需求。
- `文章`：承接长文、研究综述、阶段判断和思想碎片整理稿。
- `专题`：承接系列内容与阶段性收口。
- `帮助中心`：承接稳定知识、术语、规则、安装和文档。

### 不再作为一级主导航的栏目
- `专栏`
  - 保留内容能力，但降级为创作者页、个人页或文章页中的附属关系。
- `创作者`
  - 不再作为公开站点一级入口，保留在用户中心、作者页、内容详情页中承接。
- `思想碎片`
  - 不新增一级导航，作为文章体系内的一种内容类型或标签。
- `研究综述`
  - 不新增一级导航，作为文章和专题体系内的核心模板。

### 栏目与内容类型的对应关系
- `主题`：内容聚合层
- `问题`：FAQ/具体问题层
- `文章`：研究综述/思想碎片整理/长文层

- `帮助中心`：稳定知识层
