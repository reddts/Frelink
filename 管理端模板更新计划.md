# Frelink 管理端模板更新计划（Vue-Vben-Admin 版）

## 当前推进状态

- `2026-04-02`：
  - `M0` 已完成
  - `M1` 已完成首轮基线接入
  - 已落地：
    - `admin-vben` 子工程
    - 后台登录 / 当前管理员 / 菜单 / 仪表盘 JSON 接口
    - `/admin-vben/` 静态入口
    - 新壳层下的动态菜单与旧页占位路由
  - 下一轮主目标：
    - 按 `M2` 优先级迁移 `管理员 / 管理组 / 菜单 / 配置`
    - 再推进 `文章 / 问题 / 回答 / 审核`
  - 补充校正：
    - 后台新接口最终必须独立收口到 `adminapi` 体系，不继续并入现有前台 `api` 语义
    - 新管理端必须按独立 Vue 工程长期维护，本地安装依赖、本地测试、编译产物投放 `public` 后直接运行
    - 后续迁移重点是后台业务链路重构，不是只替换界面外壳

## 1. 文档目的

本文档用于指导将当前 `Frelink` 管理端从现有 PHP 模板后台，逐步迁移到基于 `Vue-Vben-Admin` 的新管理端框架，并确保迁移过程严格遵守项目现有推进规范。

本文档重点覆盖：

- 当前后台现状盘点
- 模板选型与前置决策
- 迁移目标与非目标
- 分阶段迁移计划
- 前后端改造约束
- 菜单、权限、页面适配规则
- 测试、发布、回滚与验收流程
- 每轮推进闭环要求

本文档不包含：

- 前台页面改版
- 业务表结构的大规模重构
- 一次性替换所有旧后台页面的硬切方案
- 脱离现有 PHP 后端的全栈重写

## 2. 前置决策

### 2.1 模板选型结论

- 用户指定参考仓库：`https://github.com/vbenjs/vue-vben-admin-doc`
- 该仓库是文档仓库，不是实际业务模板仓库。
- 该仓库已于 `2024-09-30` 被归档，只读，不适合作为实施基线。
- 实际迁移基线应使用活跃主仓库：`https://github.com/vbenjs/vue-vben-admin`
- 当前公开文档为 `Vben Admin 5.x`：`https://doc.vben.pro/`

### 2.2 本项目采用策略

- 文档参考源：`vue-vben-admin-doc` 的历史内容 + `doc.vben.pro`
- 工程实施基线：`vue-vben-admin` 主仓库当前稳定版本
- 迁移目标形态：新增独立前端管理端应用，不在旧 PHP 模板里做“套皮式改造”
- 迁移方式：新旧后台并行，逐模块切换，不允许一步替换
- 服务端接口策略：后台新接口独立命名为 `adminapi`，不继续接入现有前台 `api` 体系
- 构建策略：新管理端作为独立 Vue 子工程维护，编译产物直接发布到 `public` 中运行
- 入口策略：当前阶段后台独立接口先通过 `public/adminapi.php` 提供；若后续需要无扩展名的 `/adminapi/`，需单独补服务器 rewrite

### 2.3 采用 Vue-Vben-Admin 的理由

- 支持 `Vue 3`、`Vite`、`TypeScript`
- 具备动态菜单、权限控制、主题、国际化等中后台基础能力
- 适合做独立前端后台，而不是继续加深服务端模板耦合
- 工程化能力完整，适合后续把后台改造成可持续演进的前后端分层结构

### 3.4 当前问题总结

- 后台页面渲染方式老旧，页面结构、交互逻辑、模板样式高度耦合。
- 列表页和表单页依赖服务端 Builder，导致前端复用、状态管理和交互增强困难。
- 菜单、权限、路由仍以服务端模板为核心，不适合现代 SPA 后台。
- 弹窗、iframe、PJAX 混用，维护成本高，回归测试困难。
- 新功能继续写在旧后台，会持续扩大技术债。
- 当前缺少针对后台页面的稳定自动化测试链路。

## 4. 总目标与边界

### 4.1 总目标

构建一个新的独立管理端应用，基于 `Vue-Vben-Admin` 作为前端壳层，完成对现有后台核心功能与页面的逐步迁移，最终实现：

- 独立前端管理端工程
- 基于独立 `adminapi` 的后台页面交互
- 统一菜单、权限、路由、表格、表单、详情、上传和通知体系
- 支持旧后台与新后台双轨并行
- 支持按模块切换、按阶段回退
- 后续新功能优先落在新管理端
- 本地可安装、可测试、可编译，编译后可直接投放 `public` 运行
- 以原有模板后台功能为参照完成业务重构，而不是只做视觉替换

### 4.2 非目标

- 首阶段不删除旧后台控制器与模板
- 首阶段不重构业务表结构
- 首阶段不强行统一所有插件后台
- 首阶段不要求所有页面一次性交付
- 不允许把旧后台整个塞进新模板 iframe 里当成“迁移完成”

### 4.3 迁移原则

- 先接口化，再页面化
- 先高频模块，再低频模块
- 先双轨并行，再收口替换
- 新功能默认优先进入新管理端
- 每迁移一个模块，必须补最小验证与记录
- 页面“看起来像新模板”不算完成，必须业务链路可用
- 只改后台框架不算完成，必须同步完成对应业务链路重构
- 新后台接口、前端页面、编译发布方式必须同时成立，才算有效迁移

## 6. 技术选型与实施约束

### 6.1 前端技术约束

- 使用 `Vue 3 + TypeScript + Vite`
- 使用 `pnpm`
- 必须先在本地完成依赖安装、启动和构建验证，再进入服务器同步
- 使用统一 `request` 封装，不允许页面内散落裸请求
- 所有页面必须拆为可复用业务组件，不允许大页面堆逻辑
- 菜单、路由、权限必须统一接入，不允许页面各自实现权限判断
- 列表、表单、详情页要抽通用页面模型，避免每页重新造轮子
- 编译产物必须可直接发布到 `public/admin-vben/` 运行
- 新管理端目录必须独立维护，不与旧 PHP 模板文件混写

### 6.2 服务端技术约束

- 旧控制器保留，但新管理端不得依赖 HTML 输出
- 后台新接口命名空间统一为 `adminapi`
- 不继续复用现有面向前台/开放调用的 `api` 作为后台迁移主接口
- 新接口必须返回稳定 JSON 结构
- 分页格式必须统一
- 错误码必须统一
- 登录失效、权限不足、校验失败、业务失败要可机器识别
- 菜单和按钮权限必须能被前端消费
- 命名、目录、鉴权和返回结构都要按后台独立体系整理，避免与前台开放接口长期混用

### 6.3 接口统一返回建议

成功：

```json
{
  "code": 1,
  "msg": "ok",
  "data": {},
  "request_id": "uuid"
}
```

失败：

```json
{
  "code": 0,
  "msg": "error",
  "error_code": "PERMISSION_DENIED",
  "data": null,
  "request_id": "uuid"
}
```

分页：

```json
{
  "code": 1,
  "msg": "ok",
  "data": {
    "list": [],
    "total": 0,
    "page": 1,
    "per_page": 20
  },
  "request_id": "uuid"
}
```

## 7. 菜单、权限与页面适配规则

### 7.1 菜单迁移规则

- 以现有后台菜单权限为源数据，不重造一套独立权限语义
- 先做菜单读取适配，再做页面显示适配
- 菜单项必须标记：
  - 页面名称
  - 路由路径
  - 对应旧控制器
  - 权限标识
  - 当前迁移状态

### 7.2 权限迁移规则

- 页面权限继承现有后台权限模型
- 按钮级权限同步暴露给前端
- 前端只负责展示与拦截，最终鉴权必须仍由后端完成
- 权限缺失时，页面不得只隐藏按钮，接口也必须拒绝

### 7.3 页面适配规则

- 原列表页迁移为统一 `ListPage`
- 原新增/编辑页迁移为统一 `FormPage`
- 原详情页或审核页迁移为统一 `DetailPage`
- 原弹窗操作尽量收敛为抽屉或模态框，不继续保留 iframe 弹层模式
- 原 PJAX 行为统一收口到前端路由和状态刷新机制

## 10. 页面迁移执行清单模板

每迁移一个页面或模块，必须按以下清单执行：

1. 确认旧页面对应控制器、模板、菜单、权限节点
2. 梳理该页涉及的列表、详情、保存、删除、审核、导出接口
3. 若接口仍输出 HTML 或字段不稳定，先做接口层改造
4. 在 `admin-vben` 中建立对应路由与页面目录
5. 实现页面 UI 与交互
6. 接入权限控制
7. 完成本地验证
8. 同步服务器验证
9. 更新 `优化计划.md`
10. 更新 `CHANGELOG.md`
11. 完成中文 commit

## 11. 数据与接口改造范围

### 11.1 需要优先接口化的能力

- `adminapi` 登录与退出登录
- `adminapi` 当前管理员资料
- `adminapi` 菜单树
- `adminapi` 通用分页列表
- `adminapi` 通用详情读取
- `adminapi` 通用保存与编辑
- `adminapi` 通用状态切换
- `adminapi` 通用删除 / 批量操作
- `adminapi` 上传
- `adminapi` 首页统计与洞察数据

### 11.2 需要特别关注的旧后台行为

- Builder 自动生成字段
- iframe 弹窗打开编辑页
- PJAX 局部刷新
- 表格顶部批量操作
- 异步导出
- 原有模板中的联动字段与动态显示逻辑

### 11.3 接口改造原则

- 优先保持业务语义不变，只改输出形式
- 一次只收口一类返回结构，不同时大改多个模块
- 对高频页先做适配层，不先追求接口“绝对完美”
- 后台接口应优先从旧控制器/模型中抽业务能力，逐步沉淀为新的 `adminapi`，而不是把旧模板输出简单包一层
- 接口命名、字段、错误码、分页和权限结构要按新后台统一规范重构，避免继续背负旧命名债务

## 12. 测试与验收规范

### 12.1 本地最小验证

- 前端依赖安装成功
- 前端项目可启动
- 前端项目可构建
- 登录链路可用
- 目标页面可打开
- 列表、搜索、分页、保存、删除至少走通一次
- 控制台无阻断级报错

### 12.2 服务端最小验证

- 相关 PHP 文件 `php -l`
- 相关接口可访问
- 缓存清理或必要刷新命令执行完成
- 涉及菜单或权限时完成账号验证

### 12.3 页面验收标准

- 页面结构完整
- 权限控制正确
- 成功、失败、空态、加载态明确
- 表单校验可用
- 关键业务操作可闭环
- 与旧后台结果一致，或差异已明确记录

### 12.4 自动化测试建议

- 单元测试：
  - 公共工具函数
  - 通用表单 / 表格组件
- 集成测试：
  - 登录
  - 菜单生成
  - 权限控制
- E2E：
  - 文章管理
  - 用户管理
  - 审核流程
  - 配置保存

## 13. 发布、切换与回滚策略

### 13.1 发布策略

- 新管理端先使用独立入口部署
- 不直接覆盖旧后台入口
- 采用灰度方式让管理员先试用

### 13.2 切换策略

- 先开放测试账号
- 再开放部分高频菜单
- 最后切换默认后台入口

### 13.3 回滚策略

- 保留旧后台控制器与模板
- 保留原登录入口
- 菜单可按开关回退到旧页
- 若新管理端异常，可立即恢复旧后台为主入口

## 14. 风险清单

### 风险 1：误把归档文档仓库当作实施基线

- 影响：启动后发现目录、依赖、版本全部过时
- 应对：实施基线统一使用 `vue-vben-admin` 主仓库和 `doc.vben.pro`

### 风险 2：接口层未先收口，前端页面无法稳定推进

- 影响：页面迁移反复返工
- 应对：优先把登录、菜单、列表、表单 API 契约稳定下来

### 风险 3：新旧后台并行期维护成本过高

- 影响：同一功能维护两套页面
- 应对：新功能优先只进新管理端，旧后台只做兜底维护

### 风险 4：过早全量迁移导致交付失控

- 影响：周期过长、无法闭环
- 应对：按 P0、P1、P2 分批，先打通高频链路

### 风险 5：权限、菜单、按钮行为迁移不一致

- 影响：新后台能看不能用，或越权
- 应对：以前后端双重鉴权为硬约束，逐页验收

## 15. 严格执行的项目流程规范

### 15.1 每轮推进固定流程

后续每一轮与管理端模板迁移相关的工作，必须严格按照以下顺序执行：

1. 明确本轮唯一主目标
2. 更新或确认 `管理端模板更新计划.md` 中对应阶段
3. 完成代码改动
4. 完成本地最小验证
5. 同步服务器
6. 完成服务器最小验证
7. 更新 `优化计划.md`
8. 更新 `CHANGELOG.md`
9. 提交一次中文 commit

### 15.2 文档联动规则

- `管理端模板更新计划.md`：负责迁移主计划、阶段、范围、规则
- `优化计划.md`：负责纳入项目级推进目标与当前待办

### 15.3 完成定义

- `CHANGELOG.md` 已更新
- 已提交中文 commit

## 17. 当前结论

- 本项目适合采用 `Vue-Vben-Admin` 作为新管理端模板。
- 但实施基线不能直接使用已归档的 `vue-vben-admin-doc` 仓库。
- 正确路径是：参考旧文档思路，基于活跃的 `vue-vben-admin` 5.x 工程接入，并用新旧后台并行方式完成迁移。
- 迁移成败不取决于“模板是否套上”，而取决于登录、菜单、权限、列表、表单、审核、发布、运维等后台主链路是否真正落地。
