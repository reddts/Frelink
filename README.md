# Frelink

我把 Frelink 作为一个面向个人站长、垂直社区和团队知识库的开源知识检索与问答系统来持续演进。

我希望它不制造付费门槛、不诱导关注、也不把内容割裂成只能在站内反复跳转的碎片流，而是把问题、文章、主题、专题和帮助文档组织成一个可以持续检索、持续沉淀、持续复用的开放知识站。

## 产品定位

- 我把它定位为公开知识检索站：让访客优先看到完整内容，而不是登录墙和付费提示。
- 我把它定位为垂直主题社区：用主题聚合问题、文章、专题和帮助文档，形成稳定的知识结构。
- 我也把它定位为可二次开发的开源程序：提供 Web、移动端和 API 基础能力，方便继续扩展自动化和 AI 能力。

## 适用场景

- 我用它来做个人公开知识站
- 我也认为它适合行业垂直问答社区
- 我认为它可以作为企业内部知识库底座
- 我认为它适合开源项目文档与 FAQ 中心
- 我也把它当作围绕主题聚合内容的搜索型站点方案

## 安装方式

### 1. 环境要求
- PHP >= 7.4
- MySQL 5.7+
- Nginx/Apache（站点根目录指向 `public/`）
- PHP 扩展：`json`、`iconv`、`zip`、`curl`、`gd`、`zlib`、`openssl`

### 2. 获取代码与依赖
```bash
git clone https://github.com/reddts/Frelink.git
cd Frelink
composer install
```

### 3. 配置数据库
1. 创建数据库（建议 `utf8mb4`）。
2. 导入安装 SQL：`install/sql/install.sql`。
3. 配置数据库连接（按你的部署环境修改配置文件/环境变量）。

### 4. 安装锁与目录权限
- 安装完成后创建安装锁文件：`install/lock/install.lock`（避免重复进入安装流程）。
- 确保以下目录可写：
  - `runtime/`
  - `public/storage/`
  - `public/backup/`（如启用备份功能）

### 5. Web 服务器配置
- 站点入口：`public/index.php`
- 后台入口：`public/admin.php`
- 伪静态可参考：`public/nginx.htaccess`、项目根目录 `nginx.htaccess`

## 核心能力
- 我提供知识问答能力：提问、回答、评论、收藏、关注
- 我支持开放检索：问题、文章、主题、帮助文档统一搜索
- 我支持主题聚合：相关主题、主题合并、无限级主题
- 我支持多种内容组织形态：文章、专题、帮助、页面等
- 我保留社区治理能力：审核记录、敏感词、内容审核流程
- 我提供用户体系：用户组、权限、积分、威望、认证
- 我支持多端：网页端、移动端、API
- 我也保留扩展机制：插件系统、钩子、定时任务

## 推荐的信息架构

- 我建议首页承担搜索、推荐流、公告和核心主题入口
- 我建议问题页承接长尾搜索词和讨论入口
- 我建议文章页承接教程、案例和方案总结
- 我建议主题页作为聚合页连接问题、文章和专题
- 我建议专题页承担系列化内容整理
- 我建议帮助页承担术语、规则、文档和新手指南

## API 与自动化

- 我把 API 入口放在 `app/api/v1`，当前默认版本是 `v1`
- 详细接口说明我放在 [docs/api-v1.md](/docs/api-v1.md)
- 我当前把 API 主要用于：
  - 移动端数据接口
  - 站点集成层
  - agent 辅助巡检与运营建议输入
- 我已经落地、可直接用于自动化的能力包括：
  - 公开配置与公开内容读取
  - 搜索与热点搜索读取
  - 运营洞察埋点上报：`POST /api/Insight/track`
  - 最近窗口运营摘要、关键词、内容趋势、主题趋势、推荐建议
  - 本周执行清单：`GET /api/Insight/weekly_execution`
  - Markdown 周报导出：`GET /api/Insight/weekly_execution?format=markdown`
- 现阶段我建议把 agent 的能力限制在低风险辅助场景：
  - 内容巡检
  - 搜索词选题
  - 站点收录检查
  - 发布后 smoke test
  - 周执行清单读取与整理
  - 运维告警汇总
- 现阶段我不建议直接向 agent 开放：
  - 删除内容
  - 修改权限
  - 无审批自动正式发布
  - 无确认的生产运维操作
- 如果要接入自动化，我建议你先这样做：
  - 为 agent 单独准备低权限账号，不复用管理员账号
  - 先接只读接口，再逐步接草稿、上传和发布辅助能力
  - 为发布、推荐、置顶、删除等动作保留审计日志
  - 对登录、发布、评论、上传等接口增加限频和监控
- 我建议最少覆盖这些 smoke test：
  - `GET /api/Common/config`
  - `GET /api/Common/search`
  - `GET /api/Question/index`
  - `GET /api/Article/index`
  - `GET /api/Insight/weekly_execution`
  - `POST /api/Insight/track`
- 从当前版本开始，我对自动化相关改动的要求不只是本地可用：
  - 每轮功能改动完成后，必须同步服务器
  - 必须做服务器端最小验证，包括 `php -l`、缓存清理和相关页面/API 的 smoke test

## 本地部署环境

- 标准部署入口：
  - `bash scripts/deploy.sh show-config`
  - `bash scripts/deploy.sh sync`
  - `bash scripts/deploy.sh verify`
  - `bash scripts/deploy.sh deploy`
- 本地私有配置文件：
  - `deploy.local.json`
  - 已加入 `.gitignore`，不会进入仓库
- 配置模板与说明：
  - `deploy.local.json.example`
  - `docs/deploy.local.example.md`
- 同步排除规则：
  - `.deployignore`
- 如果私钥来自 Windows 挂载盘，建议复制到 Linux 本地路径后再使用，避免 OpenSSH 因权限过宽拒绝加载
