# 性能优化计划任务

## 目标
- 缩短首页打开时间（首字节时间 + 首屏渲染时间）
- 降低首页请求的数据库查询次数与阻塞资源体积

## 一、立即执行（高优先级）
- [ ] 生产环境关闭调试与SQL监听
  - `.env`：`APP_DEBUG=false`、`DATABASE.DEBUG=false`
  - `config/database.php`：`trigger_sql=false`
- [ ] 开启字段缓存
  - `config/database.php`：`fields_cache=true`
- [ ] 移除每请求缓存连通性测试
  - `app/common/controller/Base.php` 中 `Cache::store(...)->set('aws_cache_test')` 逻辑迁移到后台配置保存时校验

## 二、首页后端查询优化（高优先级）
- [ ] 首页小组件增加缓存（30~120秒）
  - `sidebar/announce`、`sidebar/hotTopic`、`sidebar/hotUsers`、`sidebar/hotColumn`、`common/links`
- [ ] 修复 `PostRelation::processPostList` 的 N+1 查询
  - 批量获取：是否已回答、投票状态、关注状态、回答用户
  - 用内存映射替代循环内重复 `db()->where()->value()/column()`
- [ ] 将在线状态清理逻辑移出前台请求
  - `CheckHelper::checkOnline()` 中的过期删除与更新迁移到定时任务

## 三、SQL与索引优化（中优先级）
- [ ] 减少 `ORDER BY RAND()` 使用
  - 替换为随机种子/随机ID段/预计算推荐池
- [ ] 为高频查询补充索引
  - `post_relation(status,item_type,update_time,popular_value,category_id)`
  - `answer(question_id,uid,status,agree_count)`
  - `users_follow(fans_uid,friend_uid)`
  - `topic_relation(topic_id,item_type,item_id,status)`

## 四、前端首屏优化（高优先级）
- [ ] 非关键JS改为 `defer` 或按需加载
  - `layui.all.js`、`webuploader.js`、`module.min.js` 等
- [ ] 首页仅加载必要资源
  - 不在首页使用的库从全局模板剥离
- [ ] 继续压缩与缓存策略
  - 启用 gzip/brotli
  - 静态资源设置长期缓存（带版本号）

## 五、验证与验收（必须）
- [ ] 记录改造前基线
  - 指标：TTFB、FCP、LCP、首页SQL条数、接口耗时P95
- [ ] 分阶段发布与回归
  - 每次只上线一类优化，观察24小时
- [ ] 最终验收目标
  - 首页响应时间下降 30%+
  - 首页SQL查询数量下降 40%+

## 风险与回滚
- [ ] 所有改动保留开关配置
- [ ] 异常时优先回滚资源加载策略与查询改造
